# Agent + RAG 企业 IT 知识库工单助手 项目详解

## 1. 这个项目是做什么的

这是一个面向企业 IT 支持场景的 MVP 项目，目标是把常见 IT 问题的处理流程做成一个可演示、可评测、可扩展的问答系统。

它解决的是一个很典型的业务问题：

- 员工提 IT 问题时，优先从知识库里给出标准答案
- 如果知识库没有命中，或者用户明确要求升级处理，就自动创建工单
- 同时保留评测、追踪和运行模式切换能力，方便演示和后续扩展

你可以把它概括成一句话：

> 这是一个把 RAG 检索、问答生成、升级工单和效果评测串起来的企业 IT 支持助手。

## 2. 项目适合怎么对面试官开场

建议先用 30 秒讲清楚业务价值，再展开技术实现。

### 30 秒版本

我做的是一个企业 IT 知识库工单助手。用户提问后，系统会先做 RAG 检索，如果知识库命中且置信度够高，就直接返回处理步骤；如果命中不足或者用户明确要求升级，就自动创建工单。项目同时支持离线模式和 OpenAI Compatible 模式，并带有一套内置评测机制，可以快速验证问答和工单分流是否正常。

### 2 分钟版本

这个项目的核心目标是模拟企业内部 IT 支持的一线处理流程。整体架构是 Streamlit 前端加 FastAPI 后端，后端里把文档导入、分块、向量化、检索、答案生成、工单落库和评测都拆成了独立服务。  
知识库默认存放在本地文本或 PDF 文件中，导入时会先解析文件、切分成 chunk，再写入 ChromaDB。用户提问后，系统会先做向量检索，同时叠加词法覆盖分数，得到一个更稳的综合分数。  
如果 Top1 结果达到阈值，就走回答分支；否则或者用户明确说“还是不行，帮我创建工单”，就走工单分支，把问题、原因和摘要写入 SQLite。  
为了方便演示和测试，项目支持 offline 模式，不依赖外部模型接口；如果切换到 OpenAI Compatible 模式，也可以直接接通义千问这类兼容 OpenAI API 的模型服务。除此之外，我还做了内置评测和评测历史落盘，这样每次调整阈值或模型都能快速回归验证。

## 3. 你要先掌握的项目定位

这个项目的定位是：

- 一个单 Agent 决策的 RAG 问答系统
- 一个带工单升级能力的企业 IT 支持原型
- 一个支持离线演示和真实模型切换的 MVP

这里有一个面试中很重要的边界：

- 它不是复杂的多智能体系统
- 它的“Agent”更准确地说，是一个带规则决策的支持助手
- 它不是纯 LLM 生成系统，核心依赖检索结果和置信度阈值

这点一定要诚实。这样反而更专业。

## 4. 整体架构

### 技术栈

- 前端：Streamlit
- 后端：FastAPI
- 向量库：ChromaDB
- 工单与评测存储：SQLite
- 可选模型接入：OpenAI Compatible API
- 可选链路追踪：LangSmith

### 架构分层

#### 1. 前端层

`frontend/app.py`

负责：

- 展示服务状态
- 载入示例知识库
- 上传文档并建索引
- 提问并展示回答 / 工单
- 查看已载入文档
- 运行评测并查看评测历史

#### 2. API 层

`backend/app/main.py`

负责：

- 创建 FastAPI 应用
- 初始化配置和服务实例
- 暴露 `/chat`、`/ingest`、`/status`、`/tickets`、`/evaluate/samples` 等接口

#### 3. 领域服务层

位于 `backend/app/services/`

主要服务：

- `document_loader.py`：文档读取，支持 txt、md、pdf
- `chunker.py`：分块
- `embedder.py`：向量化，支持离线和 OpenAI Compatible
- `vector_store.py`：向量存储、检索、来源汇总
- `responder.py`：答案生成和工单摘要生成
- `agent.py`：问答还是建单的核心决策
- `tickets.py`：工单创建与查询
- `evaluator.py`：内置评测
- `evaluation_history.py`：评测历史持久化

#### 4. 基础设施层

- `config.py`：统一配置
- `database.py`：SQLite 连接和表初始化
- `logging_config.py`：日志
- `tracing.py`：LangSmith 的可选接入

## 5. 一条用户请求是怎么走完的

这里是面试最值得讲清楚的主链路。

### 5.1 知识库导入链路

用户点击“载入示例知识库”或上传文档后，系统会：

1. 读取文件内容
2. 按 chunk size 和 overlap 切分文本
3. 为每个 chunk 生成 `chunk_id`
4. 调用 embedder 生成向量
5. 把文本、metadata、embedding 一起写入 ChromaDB

对应代码：

- `backend/app/services/document_loader.py`
- `backend/app/services/chunker.py`
- `backend/app/services/ingestion.py`
- `backend/app/services/vector_store.py`

### 5.2 提问链路

用户提交问题后，请求进入 `/chat`，然后由 `SupportAgent.chat()` 处理。

主流程：

1. 对问题做向量检索
2. 取回 TopK chunk
3. 计算综合分数
4. 判断是否低置信度
5. 判断用户是否明确要求升级
6. 决定返回答案还是创建工单

对应代码：

- `backend/app/main.py`
- `backend/app/services/agent.py`
- `backend/app/services/vector_store.py`
- `backend/app/services/responder.py`

## 6. RAG 是怎么实现的

### 6.1 文档分块

分块逻辑在 `chunker.py`。

默认参数：

- `CHUNK_SIZE = 700`
- `CHUNK_OVERLAP = 120`

设计思路：

- 先做基础规范化，统一换行
- 文本短于 chunk size 时不切
- 长文本按长度切分
- 如果 chunk 末尾附近有换行，则优先在较自然的位置断开
- 使用 overlap 降低跨块信息丢失

这部分你可以对面试官说：

> 我没有简单按固定字数硬切，而是做了 overlap 和换行优先截断，这样更适合 FAQ 类知识文档。

### 6.2 向量化

项目支持两种 embedding 模式。

#### 离线模式

`LocalHashEmbedder`

特点：

- 不依赖外部模型
- 用 token 哈希生成确定性向量
- 适合本地演示、测试和离线环境

优点：

- 可运行性强
- 不需要 API Key
- 测试结果稳定

缺点：

- 语义表达能力不如真实 embedding 模型

#### OpenAI Compatible 模式

`OpenAICompatibleEmbedder`

特点：

- 请求 `/embeddings`
- 可以接兼容 OpenAI API 的模型平台
- 适合真实效果验证

### 6.3 检索打分

这是这个项目比较值得讲的点。

项目不是只看向量距离，而是把两个分数混合起来：

- `dense_score`：由向量距离换算而来
- `lexical_score`：词法覆盖率分数

最终分数公式：

`score = 0.45 * dense_score + 0.55 * lexical_score`

对应代码在 `backend/app/services/vector_store.py`。

这样设计的原因：

- 纯向量检索在小知识库里有时会语义接近但词面偏离
- 企业 IT 问题里有很多固定术语，比如 VPN、Git、打印机、密码错误
- 加入词法覆盖可以降低“看起来像，但其实不对”的误召回

### 6.4 中文检索优化

`text_matching.py` 做了几件很实用的事：

- 同义表达归一化
- 停用短语过滤
- 中文 token、2-gram、3-gram 混合切分
- 英文和下划线词单独保留

这是一个很适合讲工程思维的点：

> 因为这个项目面对的是中文 IT 问题，而且知识文档比较短，我没有直接依赖第三方分词器，而是做了轻量、确定性强的中文词法特征处理，这样部署成本低，也更便于离线演示。

## 7. “Agent” 决策到底做了什么

核心逻辑在 `backend/app/services/agent.py`。

它做了两层判断：

### 7.1 低置信度判断

只要满足以下任一条件，就认为不适合直接回答：

- 没有检索结果
- `top_score < RAG_SCORE_THRESHOLD`
- `top_lexical_score < RAG_LEXICAL_SCORE_THRESHOLD`

默认阈值：

- `RAG_SCORE_THRESHOLD = 0.22`
- `RAG_LEXICAL_SCORE_THRESHOLD = 0.2`

### 7.2 显式升级判断

如果用户输入包含类似下面的表达，就直接升级：

- “试过了还是不行”
- “创建工单”
- “转人工”
- “升级处理”

所以它的决策本质是：

> 检索置信度不足，或者用户明确要求升级，就创建工单；否则就基于检索结果生成答案。

面试时可以把它描述为：

> 这是一个规则增强的单 Agent 决策器，而不是完全依赖 LLM 的黑盒路由。

## 8. 回答是怎么生成的

`SupportResponder` 负责这件事。

### 两种模式

#### 1. 启发式回答

离线模式默认走这个分支。

它会：

- 取 Top1 chunk 作为主参考
- 尝试提取处理步骤
- 生成结构化回答
- 附带补充参考文档
- 提示用户如果仍然失败，可以升级建单

优点：

- 可控
- 不依赖模型
- 演示稳定

#### 2. LLM 回答

如果配置了聊天模型，就调用 OpenAI Compatible Chat 接口。

Prompt 的约束比较明确：

- 只能基于提供的知识片段回答
- 不允许编造
- 优先输出可执行步骤
- 最后提示失败可升级工单

这里你可以讲：

> 我保留了 LLM 分支，但同时设计了启发式兜底。即使外部模型失败，系统仍然能工作。

这是比较稳的工程说法。

## 9. 工单能力是怎么做的

工单相关逻辑在：

- `backend/app/services/tickets.py`
- `backend/app/database.py`

SQLite 中有 `tickets` 表，字段包括：

- `ticket_no`
- `user_question`
- `reason`
- `summary`
- `created_at`

工单号规则：

- 前缀为当天日期，如 `IT-20260426-001`
- 同一天内按顺序递增

创建工单时，系统会记录：

- 用户问题
- 升级原因
- 一条工单摘要

这个摘要既可以由 LLM 生成，也可以由启发式逻辑生成。

面试时这部分可以强调：

> 这个项目不是只停留在“回答问题”，而是把回答失败后的业务闭环补上了，这对企业支持场景更真实。

## 10. 为什么要做评测

这是项目里非常加分的一块。

很多 RAG Demo 只做到“能跑”，但这个项目额外做了“能验证”。

### 评测能力

在 `backend/app/services/evaluator.py` 里定义了一组默认测试用例，包括：

- VPN 问题应该命中 `vpn_troubleshooting.txt`
- 打印机问题应该命中 `printer_failure.txt`
- 邮箱密码错误应该命中 `email_login_reset.txt`
- Git push 权限问题应该命中 `git_access_issue.txt`
- 明确升级请求应该建工单
- 未知问题应该建工单

### 评测指标

每条评测用例都会校验：

- 实际模式是不是 `answer` 或 `ticket`
- 期望命中的文档是否正确
- 是否返回了引用

### 评测历史

评测结果会存进 SQLite 的 `evaluation_runs` 表中，支持：

- 查看批次列表
- 按 `run_id` 查看完整详情
- 重置时可选择清空评测历史

### 这块该怎么讲

> 我在项目里加入了内置评测和评测历史，是为了避免只靠人工试几个问题来判断效果。这样每次修改 chunk 策略、阈值或者模型时，都能快速回归验证主链路有没有退化。

这句话很适合面试。

## 11. 配置与运行模式

配置集中在 `backend/app/config.py`，默认从 `.env` 和 `.env.local` 加载。

### Offline 模式

配置：

- `CHAT_PROVIDER=offline`
- `EMBEDDING_PROVIDER=offline`

特点：

- 不依赖外部 API
- 本地可演示
- 测试稳定

### OpenAI Compatible 模式

配置：

- `CHAT_PROVIDER=openai_compatible`
- `EMBEDDING_PROVIDER=openai_compatible`
- `MODEL_API_BASE`
- `MODEL_API_KEY`
- `CHAT_MODEL_NAME`
- `EMBEDDING_MODEL_NAME`

特点：

- 可以接真实模型
- 更适合验证真实语义能力

面试时你可以说：

> 我特意保留了离线模式和真实模型模式两套运行方式，这样既能保证项目开箱即跑，也能在需要时切到真实模型验证效果。

## 12. 前端页面都在干什么

前端虽然是 Streamlit，但逻辑完整，足够支撑演示。

页面主要分为几块：

- 服务状态
- 知识库管理
- 已载入文档
- 提问
- 内置评测
- 评测历史
- 最近工单

其中比较重要的两个展示项：

- `知识文档`：当前知识库文件数
- `向量文档`：当前 chunk 数量

这两个值很容易混淆，面试中如果你能主动讲清楚，说明你真的理解系统数据结构。

## 13. 对外接口怎么讲

核心 API：

- `GET /status`：查看系统状态
- `GET /health`：健康检查
- `GET /documents`：查看已载入文档
- `POST /ingest`：上传文档并建索引
- `POST /ingest/samples`：导入示例知识库
- `POST /chat`：提问
- `GET /tickets`：查看工单
- `POST /reset`：重置系统状态
- `POST /evaluate/samples`：运行内置评测
- `GET /evaluations`：查看评测历史
- `GET /evaluations/{run_id}`：查看某次评测详情

建议你面试时不要把接口逐个背诵，而是讲成三类：

- 运行状态类：`/status`、`/health`
- 知识库管理类：`/ingest`、`/documents`、`/reset`
- 核心业务类：`/chat`、`/tickets`、`/evaluate/samples`

## 15. 这个项目的亮点

面试时建议重点讲这几条。

### 亮点 1：不是单纯 RAG，而是补了工单闭环

大多数 Demo 只做到问答，这个项目把“答不上来怎么办”也做了。

### 亮点 2：支持离线运行，适合演示和测试

离线 embedding 和启发式 responder 让系统不依赖外部模型也能跑通。

### 亮点 3：检索分数不是只看向量，而是混合词法特征

这让小知识库场景更稳，尤其适合中文 FAQ 和企业术语问题。

### 亮点 4：内置评测和评测历史

这部分很像一个小型回归测试体系，不只是“看起来能用”。

### 亮点 5：工程拆分比较清晰

文档解析、向量化、检索、生成、工单、评测都拆成了独立服务，便于替换和扩展。

## 16. 这个项目的不足，要怎么诚实回答

这部分非常重要。不要把项目吹成生产级。

### 不足 1：Agent 决策主要还是规则驱动

虽然叫 Agent，但本质还是单 Agent + 规则决策，不是复杂推理型 Agent。

### 不足 2：离线 embedding 只是演示级方案

哈希向量保证了可运行性，但语义能力有限。

### 不足 3：知识库规模还偏小

当前样例文档只有 5 个，更偏 MVP 和面试演示。

### 不足 4：工单流转还不完整

目前只有工单创建和展示，还没有状态机、处理人、SLA、通知等能力。

### 不足 5：缺少更强的权限和多租户设计

现在是单实例、单知识库的简化版本。

### 推荐回答方式

> 这个项目我把它定位成可演示、可评测、可扩展的 MVP，而不是生产级系统。所以我优先保证了链路闭环、可运行性和可测试性。像更强的语义检索、工单状态流转、权限控制和多租户，是下一阶段更适合补的能力。

## 18. 面试里最可能问到的 10 个问题

### 1. 为什么用 Streamlit，而不是前后端完全分离框架

因为这是一个面试版 MVP，重点是快速验证业务链路。Streamlit 可以快速搭建交互界面，把精力集中在后端核心逻辑上。

### 2. 为什么选 ChromaDB

因为它轻量、易落地、支持本地持久化，适合小型知识库和 Demo 场景。

### 3. 为什么要同时用向量分数和词法分数

因为小规模 FAQ 型知识库容易出现语义接近但词面不对的误召回。混合打分可以提升稳定性。

### 4. 为什么要做离线模式

因为面试演示和本地开发时，不能依赖外部 API 的可用性。离线模式让系统具备开箱即跑能力。

### 5. 为什么要加评测

因为仅靠手工点几个问题不够稳。评测可以在调整阈值、chunk 策略或模型时快速回归。

### 6. 这个项目里 Agent 的价值是什么

价值不在“复杂推理”，而在于把检索结果、用户意图和业务动作做了路由决策，形成了回答和建单的业务闭环。

### 7. 如果命中了错误文档怎么办

当前做法是用混合分数和阈值控制降低误召回；后续可以增加 rerank、query rewrite 和人工反馈闭环。

### 8. 为什么工单单独落 SQLite，不直接放前端状态里

因为工单属于业务实体，需要持久化、可查询、可回放，SQLite 是一个足够轻量的本地方案。

### 9. 真实模型模式下怎么避免幻觉

关键是把回答建立在检索结果之上，并在 Prompt 中限制只能基于给定上下文回答，同时保留低置信度升级策略。

### 10. 这个项目如果上线，最大的风险是什么

最大的风险是知识库规模扩大后召回质量和阈值稳定性可能下降，所以需要更强检索和更系统的线上评测。

## 23.我的疑问

#### 1.这个项目还用了聊天模型和向量模型,它们分别是干什么的


• 有，两个模型职责完全不同：

  聊天模型

  - 负责“生成文字答案”
  - 在这个项目里，用来根据检索到的知识片段组织最终回复，或者生成工单摘要
  - 对应代码主要在 backend/app/services/responder.py

  向量模型

  - 负责“把问题和文档都转成向量”
  - 这样系统才能做相似度检索，找到最相关的知识片段
  - 对应代码主要在 backend/app/services/embedder.py 和 backend/app/services/vector_store.py

  你可以把它们简单理解成：

  - 向量模型：先帮你“找资料”
  - 聊天模型：再帮你“组织答案”

  在这个项目里的实际流程是：

  1. 先用向量模型把用户问题和知识库文档做相似度匹配
  2. 找到最相关的几个片段
  3. 如果命中足够好，就交给聊天模型生成回答
  4. 如果命中不好，或者用户要求升级，就不走正常回答，直接创建工单

  还有一个细节你面试时可以讲：

  - 这个项目支持 offline 模式
  - offline 模式下，聊天模型和向量模型都可以不用真实大模型
  - 向量模型会退化成一个本地哈希向量器
  - 聊天模型会退化成启发式回答器

  所以它不是“必须联网才能跑”的项目，这也是它的一个工程亮点。

  找最相关的知识片段；聊天模型主要负责生成，它拿到检索结果之后，把这些片段整理成更自然、更像人工支持的回复，或者生成工单

#### qwen-plus 做聊天模型，选 text-embedding-v4 做向量模型,那为什么在侧边栏里聊天模型写的是openai_compatible / qwen-plus,  向量模型写的是openai_compatible / text-embedding-v4

  - provider / model

  所以你看到的是：

  - 聊天模型：openai_compatible / qwen-plus

  - openai_compatible：调用方式，也就是“通过兼容 OpenAI API 的接口来接模型”
  - qwen-plus 或 text-embedding-v4：实际使用的具体模型名

  也就是说：

  - openai_compatible 不是模型
  - 它是 provider，表示这一层代码走的是兼容 OpenAI 的协议
  - 真正的聊天模型是 qwen-plus
  - 真正的向量模型是 text-embedding-v4

  对应代码在：

  - backend/app/services/providers.py:25
  - backend/app/services/responder.py:18
  - backend/app/services/embedder.py:44

  你可以把它理解成：

  - provider = “怎么接”
  - model = “接的是谁”
