项目名称
基于 Agent + RAG 的企业 IT 知识库工单助手（面试版 MVP）

一、项目定位
- 面向面试展示的单用户 Demo，不追求大而全，强调完整闭环和工程思路。
- 核心目标是实现“先知识库问答，无法解决时自动升级为工单”的真实业务流程。
- 重点展示 5 个能力：
  1. RAG 检索
  2. Agent 决策
  3. Tool Calling
  4. 可观测性
  5. 基础评测

二、项目边界
做什么
- 基于本地 IT 文档库进行问答
- 回答时附带引用来源
- 当知识库无法解决，或用户明确表示“试过了还是不行”时，调用创建工单工具
- 记录关键链路日志，并准备少量核心测试用例

不做什么
- 不做多用户和权限系统
- 不做真实 Jira、飞书、邮件联调，统一用模拟工具代替
- 不做多 Agent
- 不做复杂对话记忆
- 不做 OCR 优化和多格式复杂解析

三、最终技术栈

1. 前端层
- Streamlit
- 单页面应用
- 功能只保留：
  - 文档上传
  - 用户提问
  - 展示回答
  - 展示引用来源
  - 展示工单创建结果

2. 服务层
- FastAPI
- 提供统一接口：
  - `/chat`：问答主入口
  - `/ingest`：导入文档并构建向量索引
  - `/tickets`：查看模拟工单记录
- 作用：
  - 把 UI、Agent、RAG、工具调用解耦
  - 方便面试时说明系统分层
  - 后续若替换前端，不影响后端逻辑

3. Agent 层
- 单 Agent
- 不使用复杂多步 ReAct，采用“轻量路由决策”模式
- 决策规则：
  - 条件 1：若检索结果为空，或 Top1 相似度低于阈值，则创建工单
  - 条件 2：若用户明确表达“试过了不行”“还是没解决”“无法处理”等升级意图，则创建工单
  - 其他情况：返回 RAG 检索答案
- 输出要求：
  - 问答分支：返回答案 + 引用来源 + 下一步建议
  - 工单分支：返回创建原因 + 模拟工单号 + 工单摘要

4. RAG 层
- 知识源：
  - 3 到 5 个自己编写的 IT 支持文档
  - 示例主题可以包括：
    - VPN 连接失败
    - 邮箱无法登录
    - 打印机故障
    - Git 权限问题
    - 公司 Wi-Fi 配置
- 文档解析：
  - 以纯文本或简单 PDF 为主
  - 第一版优先保证稳定，不引入 OCR
- 切分策略：
  - 固定长度切分
  - 建议 chunk size：500 到 800 字符
  - overlap：100 到 150 字符
- 向量化：
  - 通义千问 Embedding
- 向量库：
  - ChromaDB
  - 使用本地持久化目录
- 检索策略：
  - Top-K 检索，建议 K=3
  - 读取 Top1 分数作为是否命中知识库的判断依据
- 输出要求：
  - 返回答案时必须附带命中文档名和片段摘要

5. 工具层
- 仅保留 1 个工具：`create_ticket`
- 工具行为：
  - 输入：用户问题、检索失败原因、上下文摘要
  - 输出：模拟工单号，例如 `IT-20260418-001`
- 说明：
  - 不联真实工单系统，避免环境和鉴权成本
  - 但接口和返回结构按真实系统设计，方便后续扩展

6. 大模型层
- 只保留 1 个主模型
- 统一使用通义千问生态：
  - Chat Model：负责问答生成、路由判断、工单摘要生成
  - Embedding Model：负责向量化
- 原因：
  - 降低供应商切换成本和配置复杂度
  - 保持演示稳定
  - 避免面试时被追问多模型切换策略

7. 可观测性层
- LangSmith
  - 记录关键链路：
    - 用户输入
    - 检索结果
    - Agent 决策
    - 工具调用
    - 最终输出
- Python logging
  - 记录错误日志、接口调用、关键分支
- 测试用例
  - 准备 5 个核心测试样例
  - 至少覆盖：
    - 正常命中文档并回答
    - 问题无命中时创建工单
    - 用户说“试过了还是不行”时创建工单
    - 回答必须带引用来源
    - 工单工具返回正确格式结果

8. 数据层
- 本地知识库文件
  - 保存原始文档
- ChromaDB 持久化目录
  - 保存向量索引
- SQLite
  - 保存模拟工单记录
  - 保存测试结果或简要运行记录
- 说明：
  - 不保存复杂多轮对话历史
  - 只保留必要的数据落盘，保证结构清晰

四、整体架构

Streamlit 前端
    ↓
FastAPI 服务层
    ↓
单 Agent 路由
    ├─ 命中知识库：RAG 问答 + 引用来源
    └─ 未命中或用户要求升级：create_ticket 工具
    ↓
LangSmith + logging 记录链路
    ↓
SQLite / ChromaDB / 本地文档落盘

五、核心业务流程

流程 1：知识库问答
1. 用户在 Streamlit 输入问题
2. FastAPI 调用检索模块查询 ChromaDB
3. Agent 判断检索结果是否足够可靠
4. 若命中，则调用模型生成答案
5. 返回答案、引用来源、下一步建议
6. 记录 trace 和日志

流程 2：自动创建工单
1. 用户提问后，检索结果为空或分数低于阈值
2. 或者用户明确表示“试过了还是不行”
3. Agent 调用 `create_ticket`
4. 系统生成模拟工单号和问题摘要
5. 前端展示工单创建结果
6. 工单记录写入 SQLite

六、MVP 完成标准
- 可以上传或加载 3 到 5 个示例文档
- 可以完成至少 1 轮知识库问答
- 回答中包含引用来源
- 至少 2 类情况会触发工单创建
- 工单号可以落盘保存
- LangSmith 中可以看到完整调用链
- 5 个核心测试用例可手动复现

七、面试时的讲述重点
- 为什么不用“纯大模型直答”
  - 因为企业知识问答需要基于内部文档，RAG 可以降低幻觉并给出引用来源
- 为什么不用多 Agent
  - 因为 MVP 阶段优先保证流程清晰、调试成本低、可解释性强
- 为什么只保留一个工具
  - 因为一个真实动作已经足够展示 Agent 的决策和 Tool Calling 能力
- 为什么要有 LangSmith 和测试用例
  - 因为 Agent 项目难点不只是生成答案，而是可调试、可评估、可复现
- 为什么要加 FastAPI
  - 因为这样更像真实系统，前端和核心逻辑分离，后续扩展成本更低

八、最终结论
这是一个范围可控、能在短时间内做完、同时又能体现 Agent + RAG 核心价值的面试项目。

项目的亮点不在于堆很多组件，而在于把下面这条链路做完整：
文档入库 -> 检索召回 -> Agent 判断 -> 回答或创建工单 -> 结果落盘 -> 链路可追踪 -> 用例可验证
