# 企业 IT 知识库工单助手项目详细介绍

## 1. 项目一句话介绍

这是一个面向企业 IT 支持场景的 `单 Agent + RAG + 工单升级` 项目。  
用户提问后，系统会先从知识库中检索答案；如果知识命中且置信度足够高，就直接返回处理步骤和引用来源；如果命中不足，或者用户明确要求升级处理，系统就会自动创建工单并落库。

你可以把它概括成一句话：

> 这是一个把 RAG 检索问答、Agent 决策、工单升级和评测闭环串起来的企业 IT 支持助手。

---

## 2. 项目解决的业务问题

这个项目解决的是一个非常真实的企业内部场景：

- 员工遇到 VPN、邮箱、打印机、Git 权限这类常见 IT 问题时，希望系统优先给出标准处理步骤。
- 如果知识库解决不了，或者员工已经试过仍然失败，系统不应该继续“硬答”，而是应该自动升级成工单。
- 除了能演示，还需要能评估系统效果，验证检索命中率和工单分流逻辑是否正常。

所以这个项目的价值，不只是“会回答问题”，而是补上了 `回答失败后的业务闭环`。

---

## 3. 项目定位

这个项目更适合被定义为：

- 一个面向企业 IT 支持场景的 RAG 问答系统
- 一个带工单升级能力的单 Agent 决策系统
- 一个支持离线演示、模型切换和评测回归的 MVP 项目

这里有一个面试时很重要的边界：

- 它不是复杂的多 Agent 系统
- 它的 Agent 重点不在自主推理，而在 `路由决策` 和 `工具调用`
- 它不是纯 LLM 对话系统，而是一个以检索结果为核心依据的业务系统

面试里这样说会更专业：

> 这个项目里的 Agent，本质上是一个规则增强的单 Agent 决策器，负责在“知识库回答”和“创建工单”之间做业务路由。

## 5. 核心业务流程

这个项目最值得讲清楚的是两条链路：`知识入库链路` 和 `问答升级链路`。

### 5.1 知识入库链路

当用户点击“载入示例知识库”或上传文档后，系统会依次执行：

1. 读取文档内容
2. 按 chunk 规则切分文本
3. 为每个 chunk 生成唯一 `chunk_id`
4. 调用 embedding 服务生成向量
5. 将文本、元数据、向量一起写入 ChromaDB

涉及文件：

- `backend/app/services/document_loader.py`
- `backend/app/services/chunker.py`
- `backend/app/services/ingestion.py`
- `backend/app/services/vector_store.py`

### 5.2 问答升级链路

当用户提交问题后，请求进入 `POST /chat`，然后由 `SupportAgent.chat()` 处理。

主流程如下：

1. 对用户问题做向量检索
2. 获取 TopK 候选 chunk
3. 计算每个候选的综合分数
4. 判断是否低置信度
5. 判断用户是否明确要求升级
6. 决定返回答案还是创建工单

这个链路的核心意义在于：

- 不是所有问题都强行回答
- 系统会根据检索质量决定是否应该升级处理
- 这样比纯 LLM 问答更稳，更符合企业业务场景

## 6. RAG 是怎么实现的

### 6.1 文档读取

`DocumentLoader` 支持：

- `.txt`
- `.md`
- `.pdf`

文档为空时会直接报错，避免把无效内容写入知识库。  
PDF 解析依赖 `PyMuPDF`，如果环境没有安装，也会明确报错。

### 6.2 文本切分

切分逻辑在 `backend/app/services/chunker.py`。

默认参数：

- `CHUNK_SIZE = 700`
- `CHUNK_OVERLAP = 120`

设计要点：

- 先统一换行格式
- 文本较短时不切分
- 长文本按固定大小切分
- 优先在接近自然换行的位置截断
- 使用 overlap 降低跨块信息丢失

面试里可以这样说：

> 我没有简单按固定字符数硬切，而是做了 overlap 和换行优先截断，这样对 FAQ 和 IT 支持类文档更友好。

### 6.3 向量化

项目支持两种 embedding 模式：

#### 1. 离线模式

- `EMBEDDING_PROVIDER=offline`
- 不依赖外部 API
- 适合本地演示和自动化测试

#### 2. OpenAI Compatible 模式

- `EMBEDDING_PROVIDER=openai_compatible`
- 调用兼容 OpenAI 的 `/embeddings` 接口
- 适合接入真实模型服务，做效果验证

### 6.4 检索打分

项目不是只看向量相似度，而是用了 `混合打分`。

在 `backend/app/services/vector_store.py` 中：

- `dense_score`：由向量距离转换得到
- `lexical_score`：词法覆盖率分数

最终得分公式：

```text
score = 0.45 * dense_score + 0.55 * lexical_score
```

这样设计的原因是：

- 企业 IT 场景里有很多固定术语，比如 `VPN`、`Git`、`邮箱密码错误`
- 纯向量检索可能语义接近，但词面不准确
- 引入词法覆盖后，可以减少“看起来像，但其实不是”的误召回

这部分是面试里非常值得讲的亮点。

### 6.5 中文词法优化

`text_matching.py` 对中文检索做了轻量优化：

- 同义表达归一
- 停用短语过滤
- 中文 token 和 n-gram 混合处理
- 英文单词和下划线词保留

这个点体现的是工程取舍：

> 面向中文 IT 短文本场景，我没有一上来引入很重的分词体系，而是做了轻量、确定性更强的词法处理，部署更简单，也更适合离线演示。

## 7. Agent 决策逻辑

核心逻辑在：`backend/app/services/agent.py`

这个 Agent 做了两层判断。

### 7.1 低置信度判断

只要满足以下任意条件，就认为不适合直接回答：

- 没有检索结果
- `top_score < RAG_SCORE_THRESHOLD`
- `top_lexical_score < RAG_LEXICAL_SCORE_THRESHOLD`

默认阈值：

- `RAG_SCORE_THRESHOLD = 0.22`
- `RAG_LEXICAL_SCORE_THRESHOLD = 0.2`

### 7.2 显式升级判断

如果用户输入中包含以下表达，就直接走升级分支：

- “试过了不行”
- “试过了还是不行”
- “还是不行”
- “没解决”
- “无法处理”
- “创建工单”
- “帮我建工单”
- “转人工”
- “升级处理”

所以它的本质逻辑是：

> 检索不可靠，或者用户明确要求升级，就创建工单；否则基于知识库结果生成答案。

这也是这个项目的 Agent 最核心的价值：`做业务决策，而不是自由聊天。`

---

## 8. 回答生成逻辑

核心文件：`backend/app/services/responder.py`

系统支持两种回答模式。

### 8.1 启发式回答

在离线模式下，默认走启发式逻辑：

- 取 Top1 chunk 作为主要依据
- 尝试提取其中的处理步骤
- 组织成结构化回答
- 附带补充参考文档
- 提示用户如果仍不行可以升级成工单

优点：

- 稳定
- 可控
- 不依赖外部模型

### 8.2 LLM 回答

如果配置了聊天模型，就走 OpenAI Compatible Chat 分支。

Prompt 约束比较明确：

- 只能基于给定知识片段回答
- 不允许编造
- 优先输出可执行步骤
- 最后提醒用户可升级工单

这个设计体现了一个很好的工程思路：

> 我保留了 LLM 生成能力，但也设计了启发式兜底。即使外部模型不可用，系统仍然能工作。

## 9. 工单能力是怎么做的

相关文件：

- `backend/app/services/tickets.py`
- `backend/app/database.py`

SQLite 中有 `tickets` 表，字段包括：

- `ticket_no`
- `user_question`
- `reason`
- `summary`
- `created_at`

### 9.1 工单号规则

工单号格式为：

```text
IT-YYYYMMDD-001
```

例如：

```text
IT-20260510-001
```

同一天内按顺序递增。

### 9.2 创建工单时会记录什么

系统会落库以下信息：

- 用户问题
- 升级原因
- 工单摘要
- 创建时间

工单摘要既可以由 LLM 生成，也可以由启发式逻辑生成。

这一块在面试时可以强调：

> 这个项目不是停留在“回答问题”，而是把回答失败后的升级动作真正做成了业务闭环。

## 10. 评测闭环

这是这个项目很加分的一部分。

很多 RAG Demo 只做到“能跑通”，但这个项目额外做了“能验证”。

核心文件：`backend/app/services/evaluator.py`

### 10.1 内置评测样例

项目内置了几类默认测试用例：

- VPN 问题应命中 `vpn_troubleshooting.txt`
- 打印机问题应命中 `printer_failure.txt`
- 邮箱密码错误应命中 `email_login_reset.txt`
- Git 权限问题应命中 `git_access_issue.txt`
- 明确升级请求应创建工单
- 未知问题应创建工单

### 10.2 评测维度

每条用例会校验：

- 实际模式是否符合预期
- 命中文档是否符合预期
- 回答分支是否带引用来源

### 10.3 评测历史

评测结果会保存到数据库，前端支持查看历史批次和详情。

这部分的面试表达可以是：

> 我在项目里补了一套轻量评测能力，用来验证知识命中和工单分流是否正常，避免项目只能靠手工演示。

## 11. 项目的工程化亮点

这个项目的亮点不在于“堆了很多组件”，而在于把一条 AI 业务链路做完整了。

可以重点总结成以下几点：

### 11.1 业务闭环完整

不是只有问答，而是做到了：

`知识入库 -> 检索召回 -> Agent 决策 -> 回答或建单 -> 数据落库 -> 评测验证`

### 11.2 决策可解释

不是黑盒回答，而是基于：

- 综合检索分数
- 词法阈值
- 用户升级意图

做出明确可解释的分支决策。

### 11.3 支持离线演示

项目支持 offline 模式：

- 无需依赖外部模型
- 更适合本地演示
- 自动化测试更稳定

### 11.4 支持真实模型切换

同时也支持 OpenAI Compatible 接口：

- 可以平滑切换到真实模型
- 不需要改动主流程
- 只要修改环境变量即可

### 11.5 有评测意识

有默认用例、有历史记录、有通过率统计。  
这说明项目不只是“能展示”，还考虑了“能验证”和“能回归”。

## 12. 面试时如何介绍这个项目

### 12.1 30 秒版本

我做的是一个企业 IT 知识库工单助手。用户提问后，系统会先做 RAG 检索，如果知识库命中且置信度足够高，就直接返回处理步骤和引用来源；如果命中不足，或者用户明确要求升级，就自动创建工单并落库。项目同时支持离线模式和 OpenAI Compatible 模式，并且内置了一套评测机制，用来验证问答和工单分流是否正常。

### 12.2 2 分钟版本

这个项目模拟的是企业内部 IT 支持的一线处理流程。我的目标不是做一个泛化聊天机器人，而是把“知识库问答 + 升级工单”这条真实业务链路完整打通。整体架构上，前端使用 Streamlit，后端使用 FastAPI，向量库使用 ChromaDB，工单和评测历史使用 SQLite 持久化。  

在知识入库阶段，系统会读取 txt、md、pdf 文档，按 chunk 切分后生成 embedding 并写入向量库。用户提问后，后端会先做向量检索，同时叠加词法覆盖率，得到更稳定的综合分数。然后 Agent 根据 Top1 的得分阈值以及用户是否明确要求升级，在“直接回答”和“创建工单”之间做决策。回答分支会返回处理步骤和引用来源，工单分支会生成工单号、升级原因和摘要，并写入数据库。  

另外，我还做了一套内置评测，验证像 VPN、打印机、邮箱、Git 权限问题是否能正确命中文档，而未知问题或明确升级请求是否会进入工单分支。这样项目不仅能演示，还能做回归验证，整体更接近真实可落地的 AI 应用。
